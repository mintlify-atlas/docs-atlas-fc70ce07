---
title: "Detonation Correlation"
description: "How Threatest uses UUIDs to tie every alert back to the exact detonation that caused it."
---

When Threatest runs a scenario it needs to answer a precise question: did **this specific execution** of an attack technique produce an alert? Not an alert from a previous test run, not an alert from another team member's activity, and not a coincidental alert generated by unrelated traffic — but an alert from this exact detonation, right now.

Threatest solves this with **UUID-based correlation**.

---

## How it works

### Step 1 — UUID generation

Every call to `Detonate()` begins by generating a fresh UUID. Each detonator uses either `github.com/hashicorp/go-uuid` or `github.com/google/uuid` to produce a cryptographically random identifier that is unique across all concurrent and historical test runs.

```go
// Local / SSH detonators
id, _ := uuid.GenerateUUID()

// AWS CLI / AWS detonator
detonationUuid := uuid.New()

// Stratus Red Team — UUID is provided by Stratus itself
executionId := stratusRunner.GetUniqueExecutionId()
```

### Step 2 — UUID injection

The UUID is embedded into the detonation in a way that will be visible in the telemetry data the security platform collects. The injection mechanism differs per detonator because each one operates at a different layer of the stack.

### Step 3 — UUID-based matching

After detonation the runner passes the UUID to every registered `AlertGeneratedMatcher`:

```go
// pkg/threatest/runner.go
hasAlert, err := assertion.HasExpectedAlert(detonationUid)
```

The matcher searches the security platform for alerts that (a) match the expected rule name and (b) contain the UUID somewhere in their attributes. For Datadog Security Signals this is a raw string search over the JSON-marshalled signal attributes:

```go
// pkg/threatest/matchers/datadog/datadog.go
func (m *DatadogAlertGeneratedAssertion) signalMatchesExecution(
    signal datadogV2.SecurityMonitoringSignal,
    uid string,
) bool {
    buf, _ := json.Marshal(signal.Attributes.Custom)
    rawSignal := string(buf)
    return strings.Contains(rawSignal, uid)
}
```

## UUID injection per detonator

<AccordionGroup>
  <Accordion title="Local Command & SSH Command detonators">
    Both command-based detonators use `FormatCommand` to wrap the raw attack command before execution:

    ```go
    // pkg/threatest/detonators/utils.go
    func FormatCommand(rawCommand string, detonationUuid string) string {
        return fmt.Sprintf(
            `cp /bin/bash /tmp/%[1]s; (/tmp/%[1]s -c %[2]s || true) && rm /tmp/%[1]s`,
            detonationUuid, shellescape.Quote(rawCommand),
        )
    }
    ```

    This copies `/bin/bash` to `/tmp/<uuid>` and uses that copy as the interpreter for the attack command. The UUID therefore appears as the **parent process name** in the process tree at the time the attack runs.

    Security agents that record process lineage — such as Datadog Cloud Workload Security — will include this process name in the event data. The matcher then finds the UUID inside the signal attributes and confirms it came from this detonation.

    After the command finishes the temporary binary is deleted, leaving no persistent artefact.

    **Applies to:** `LocalCommandExecutor`, `SSHCommandExecutor`
  </Accordion>

  <Accordion title="Stratus Red Team detonator">
    Stratus Red Team assigns a unique execution ID to every detonation internally. Threatest retrieves it after detonation:

    ```go
    // pkg/threatest/detonators/stratus_red_team.go
    executionId := stratusRunner.GetUniqueExecutionId()
    return executionId, nil
    ```

    Stratus injects this ID into the AWS SDK's `User-Agent` header for every AWS API call it makes during the attack. The User-Agent is recorded in AWS CloudTrail, which is ingested by Datadog Cloud SIEM. The matcher then finds the execution ID in the signal's raw attributes.

    **Applies to:** `StratusRedTeamDetonator`
  </Accordion>

  <Accordion title="AWS CLI detonator">
    The AWS CLI reads the `AWS_EXECUTION_ENV` environment variable and appends its value to the `User-Agent` header of every request. Threatest sets this variable to `threatest_<uuid>` before running the script:

    ```go
    // pkg/threatest/detonators/aws_cli_detonator.go
    cmd.Env = append(cmd.Env, "AWS_EXECUTION_ENV=threatest_"+detonationUuid.String())
    ```

    Every AWS CLI call in the script will therefore include the UUID in the User-Agent, which CloudTrail records. Datadog Cloud SIEM surfaces this in signal attributes, and the matcher searches for the UUID string there.

    **Applies to:** `AWSCLIDetonator`
  </Accordion>

  <Accordion title="AWS detonator (programmatic)">
    The `AWSDetonator` builds a custom AWS SDK middleware that sets the `User-Agent` header on every outbound HTTP request before it is sent:

    ```go
    // pkg/threatest/detonators/aws_detonator.go
    func customUserAgentMiddleware(uniqueId uuid.UUID) middleware.BuildMiddleware {
        return middleware.BuildMiddlewareFunc("CustomerUserAgent", func(
            ctx context.Context, input middleware.BuildInput, next middleware.BuildHandler,
        ) (out middleware.BuildOutput, metadata middleware.Metadata, err error) {
            request, ok := input.Request.(*smithyhttp.Request)
            if !ok {
                return out, metadata, fmt.Errorf("unknown transport type %T", input.Request)
            }
            request.Header.Set("User-Agent", fmt.Sprintf("threatest_"+uniqueId.String()))
            return next.HandleBuild(ctx, input)
        })
    }
    ```

    The middleware is injected via `config.WithAPIOptions` at configuration load time, so it applies to every SDK call the detonation function makes regardless of which AWS service it targets.

    **Applies to:** `AWSDetonator`
  </Accordion>
</AccordionGroup>

## Why this matters

Without UUID correlation, Threatest would have to choose between two bad options:

1. **Match any alert** that matches the rule name within the polling window. This is fragile: an alert generated by a coincident real event, a different test run, or a teammate's activity would cause a false pass.
2. **Fail if any other activity generates the same alert**. This is also fragile: legitimate traffic or parallel tests would cause false failures.

UUID correlation makes the check precise. Because the UUID is embedded in the telemetry data that the security platform collects (process names, HTTP user-agents, CloudTrail events), a genuine detection of this specific execution will carry the UUID through into the alert attributes. The matcher only returns `true` when both conditions are satisfied:

- The alert matches the expected rule name (and optionally severity).
- The alert's raw attributes contain the detonation UUID string.

<Note>
  This design also means a single environment can run multiple Threatest scenarios in parallel without them interfering with each other, as long as each scenario uses a distinct UUID — which is guaranteed by construction.
</Note>

## Summary

| Detonator | Injection point | Where UUID appears |
|---|---|---|
| Local command | Process name under `/tmp/<uuid>` | Process lineage in host telemetry |
| SSH command | Process name under `/tmp/<uuid>` on remote host | Process lineage in host telemetry |
| Stratus Red Team | AWS SDK User-Agent (managed by Stratus) | CloudTrail → Cloud SIEM signal attributes |
| AWS CLI | `AWS_EXECUTION_ENV` env var → User-Agent | CloudTrail → Cloud SIEM signal attributes |
| AWS detonator | Custom SDK middleware → User-Agent | CloudTrail → Cloud SIEM signal attributes |
